## 最重要：行動原則

### 対話と実装の分離
- ユーザーの発言が相談・質問・検討の場合、絶対にコードを書くな。対話で整理しろ
- 実装に入るのはユーザーが「作って」「実装して」「やって」と明言した場合のみ
- 判断に迷ったら「実装に入っていいですか？」と確認しろ

### 計画→承認→実装
- 新しいタスクは必ず計画から始めろ。「とりあえず作る」は禁止
- 計画を提示 → ユーザー「OK」 → 実装開始
- 3ファイル以上に影響する場合はPlan Modeで設計を先に出せ

### 要件の推測と補完
- ユーザーの指示が曖昧でも、常識的に必要なものは推測して補完しろ
- 推測した内容は計画に明記し、ユーザーに確認してから実装

### 自律判断の基準
- サブエージェントの利用は自分で判断。ユーザーに聞くな
- DB設計→db-designer / 設計判断→architect / 調査→investigator / レビュー→reviewer / バグ修正→error-fixer

### 開発ワークフロー（UIファースト）
- 新しいツールや画面を作る時は、必ずUIプロトタイプから始めろ
- 手順: 計画→承認→UIのみ作る（ダミーデータ可）→ユーザー確認→裏側実装
- 外部HTMLモックは使わない。最初から正しいスタックで作れ

## ブランチ運用ルール（必須・厳守）

### 作業開始時に必ず実行
git checkout develop && git pull origin develop
⚠️ 新しいブランチを作る前に必ず実行。スキップするとマージコンフリクトが発生する。

### 絶対に守るルール
- mainには絶対に直接pushしない
- developには直接pushしない
- 作業は必ず claude/[作業内容] ブランチで行う
- claude/ブランチにpushすれば自動でdevelopに反映される（GitHub Actions）
- 古いclaude/ブランチは絶対に再利用しない
- ユーザーが最新ブランチを指定した場合は必ずそこから開始
- 指定がない場合は作業前に必ず確認する

### ブランチの役割
- main → 本番環境（ユーザーの明示的な指示があった時のみ更新）
- develop → 確認用プレビュー（自動デプロイ対象）
- claude/xxx → 作業用（pushすると自動でdevelopに入る）

### 作業完了時（毎回必ず）
- git add -A
- git commit -m "わかりやすい日本語の変更説明"
- git push origin claude/[作業内容]
- ユーザーに報告して止まる

### コミットメッセージ
- 日本語で非エンジニアにもわかる内容（例:「顧客一覧画面を追加」）
- 技術用語だけのメッセージ禁止

2. ユーザーがGitHub上でマージ

## エラー対応の型

### 調査順序（必ずこの順番で）
1. エラーメッセージを正確に読む（推測で動くな）
2. 発生箇所を特定する（フロント / API / DB / 外部連携）
3. 直前の変更点を確認する
4. 仮説を1つ立てて検証する
5. 修正 → テスト → 確認

### ループ防止
- 同じエラー2回連続 → 止まって原因候補を整理
- 3回失敗 → 人間にエスカレーション
- 修正前に必ずgit commit（ロールバック可能に）
