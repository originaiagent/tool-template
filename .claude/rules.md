# 共通行動原則（全リポ共通・同期対象）

> tool-template から自動同期。改善提案は docs/evolution-log.md に記録 → トム承認後に反映。

---

## ブランチ運用（厳守）

作業開始時に **必ず** 実行: `git checkout develop && git pull origin develop`

- main/developには絶対に直接pushしない
- 作業は `claude/[作業内容]` ブランチのみ
- 古いclaude/ブランチは再利用しない
- main = 本番（トムの明示指示のみ） / develop = 確認用 / claude/xxx = 作業用

---

## 影響範囲の調査（変更前に必須）

**YOU MUST investigate impact before making any change.**

何かを変更する前に、必ず grep/検索で影響範囲を調べてから手を動かすこと。

- カラム名・テーブル構造の変更 → そのカラムを参照する全ファイル、RLSポリシー、ビュー、関連クエリを洗い出す
- コンポーネントの変更 → importしている全ページ・全コンポーネントを確認
- API変更 → そのAPIを呼んでいるフロントエンドのコードを確認
- 型・interfaceの変更 → その型を使っている全ファイルを確認
- 関数名・引数の変更 → 呼び出し元を全て確認

影響箇所が見つかったら、全て一緒に修正してからpushすること。
「1箇所だけ直して他は後で」は禁止。部分最適より全体最適。

---

## 自己検証（push前に必須）

**YOU MUST run verification before every push. Never skip this.**

原則: 「10人中9人がおかしいと思うことは自分で直す」

### チェックリスト
1. ビルド通るか（`npm run build` / `python -m py_compile`）
2. dev server起動して主要画面が表示されるか
3. 今回変更した機能が動作するか
4. 変更していない既存機能が壊れていないか（回帰チェック）
5. コンソール/ターミナルにエラーが出ていないか

### 問題が見つかったら
自分で修正 → 再検証。3回直せなければエスカレーション。

---

## 確認基準

### トムに聞くこと
- デザイン・UXの好み
- ビジネスロジックの判断（仕様の解釈が複数ある時）
- 本番反映（mainマージ）
- 破壊的変更（データ削除、テーブル構造の破壊的変更）
- 権限上できないこと（GitHub設定、GCPコンソール、デプロイ先ダッシュボード）

### 聞かずに自分で処理すること
- ビルドエラー、型エラー、import漏れ
- 画面が白い、ボタンが反応しない、レイアウト崩壊
- コンソールエラー、API 500エラー
- typo、不要なconsole.log
- docs/やlessons-learnedに記載済みの既知問題
