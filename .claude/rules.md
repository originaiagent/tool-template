# 共通行動原則（全リポ共通・同期対象）

> このファイルは tool-template から全リポに自動同期される。
> リポ固有の情報は CLAUDE.md に書くこと。このファイルは編集しない。
> 改善したい場合は docs/evolution-log.md に提案を記録し、トムの承認後に tool-template 経由で反映する。

---

## 対話と実装の分離

- ユーザーとの対話中にコードを変更しない
- 計画を立てる → ユーザーの承認を得る → 実装する
- 不明点があれば推測せず質問する（ただし自明なバグは自分で直す）

## ブランチ運用ルール（厳守）

### 作業開始時に必ず実行
```bash
git checkout develop && git pull origin develop
```
⚠️ **新しいブランチを作る前に必ず実行。スキップするとマージコンフリクトが発生する。**

### 絶対に守るルール
- mainブランチには絶対に直接pushしない
- developブランチには直接pushしない
- 作業は必ず `claude/[作業内容]` ブランチで行う
- claude/ブランチにpushすれば自動でdevelopに反映される
- 古いclaude/ブランチは絶対に再利用しない
- ユーザーが最新ブランチを指定した場合は必ずそこから開始
- 指定がない場合は作業前に必ず確認する

### ブランチの役割
- `main` → 本番環境（ユーザーの明示的な指示があった時のみ更新）
- `develop` → 確認用プレビュー（作業後の確認はここで行う）
- `claude/xxx` → 作業用（ここにpushすると自動でdevelopに入る）

## 自己検証（push前に必須）

### 原則: 「10人中9人がおかしいと思うことは自分で直す」

push前に以下を必ず実行し、問題があれば自分で修正してからpushする。
検証をスキップしてユーザーに「確認してください」と丸投げしない。

### ビルド検証
- `npm run build` または `pip install -r requirements.txt` が成功すること
- TypeScriptの型エラーがないこと
- lintエラーがないこと（設定がある場合）

### 動作検証
- dev serverを起動して主要画面が表示されること
- 今回変更した画面・機能が意図通り動作すること
- ボタン・リンク・フォーム等のUI要素がエラーなく反応すること
- ブラウザコンソール / ターミナルにエラーが出ていないこと

### API検証
- 新規・変更したAPIエンドポイントがレスポンスを返すこと（curlで確認）
- Supabaseクエリがエラーなく実行されること

### 回帰チェック
- 変更していない既存機能が壊れていないこと（主要画面の表示確認）

### 検証で問題が見つかった場合
1. 自分で修正する
2. 再度検証する
3. 3回修正しても直らない場合のみエスカレーション

## 人間への確認基準

### 確認が必要なもの（これだけ聞く）
- デザイン・UXの好み（色、レイアウト、文言のニュアンス）
- ビジネスロジックの判断（仕様の解釈が複数ありえる場合）
- 外部への影響（課金操作、外部API契約変更）
- 本番反映（mainへのマージ判断）
- 破壊的変更（データ削除、スキーマ破壊的変更、API互換性破壊）
- 権限上できないこと（GitHub設定、GCPコンソール、デプロイ先ダッシュボード）

### 自分で処理すべきもの（聞かない）
- ビルドエラー、型エラー、import漏れ、undefined参照
- 画面が真っ白、ボタンが反応しない、レイアウト崩壊
- コンソールエラー、API 500エラー、unhandled exception
- typo、不要なconsole.log、未使用import
- CLAUDE.mdやlessons-learnedに記載済みの既知問題

## 自己進化

### 原則: 「気づいたら直す。行動原則の核は提案→承認」

### 自由に更新していいもの（承認不要）
- `docs/lessons-learned.md` — ハマったこと、解決策を発見した時
- `docs/progress.md` — 作業完了時
- `docs/architecture.md` — 技術構成に変更があった時
- `docs/evolution-log.md` — 何かを改善した時（変更内容と理由を記録）
- `CLAUDE.md` の技術スタック部分 — 事実情報が変わった時

### 提案してトムの承認後に更新するもの
- `.claude/rules.md`（このファイル）の行動原則 → 全リポに波及するため
- `.claude/commands/*` — 作業フローが変わるため
- `.claude/agents/*` — エージェントの振る舞いが変わるため
- `.claude/settings.json` の権限 — セキュリティに関わるため

### 進化の提案フォーマット
```
【進化提案】
■ 対象ファイル: [ファイルパス]
■ 変更内容: [具体的な変更]
■ 理由: [なぜ必要か、何回同じ問題が起きたか]
■ 影響範囲: [全リポ / 特定リポのみ]
```

### 更新時の義務
- 何かを改善したら必ず `docs/evolution-log.md` に記録する
- 日付、変更内容、理由、承認状態を記載する

## エラー対応の型

### 3回ルール
- 同じエラーに3回対処して解決しない場合、エスカレーションする
- エスカレーション時は技術用語を避け、小学生でもわかる表現で状況を伝える

### エスカレーション報告
```
【エスカレーション】
■ やりたかったこと: [目的]
■ 起きていること: [症状を平易に]
■ 試したこと: [3つの試行を箇条書き]
■ 考えられる原因: [推測]
■ 必要なアクション: [トムにやってほしいこと、具体的手順付き]
```

---

## 完了報告の型

```
【完了報告】

■ 実施した作業
- [箇条書き]

■ 自己検証結果
- ビルド: ✅/❌
- 画面表示: ✅/❌
- 変更機能の動作: ✅/❌
- コンソールエラー: ✅なし/❌あり（詳細）
- 既存機能の回帰: ✅問題なし/❌問題あり（詳細）

■ 人間の確認が必要な項目（なければ「なし」）
- [具体的な判断ポイント]

■ 自己進化の記録（更新したdocsがあれば）
- [更新したファイルと内容]

■ 確認URL
- [develop環境のURL]
```
